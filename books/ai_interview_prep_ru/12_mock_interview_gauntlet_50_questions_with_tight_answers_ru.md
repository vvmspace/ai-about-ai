# Mock Interview Gauntlet: 50 вопросов и плотные ответы

К этому моменту проблема уже не в подготовке.
Проблема в исполнении.
Под давлением длинные ответы разваливаются.
Поэтому я тренирую короткие, защищаемые формулировки.

Эта глава — мой финальный репетиционный набор.
Пятьдесят вероятных вопросов.
Плотные ответы, которые я могу дать за 20–60 секунд.
У каждого ответа одно правило: claim, evidence, trade-off.

Если я начинаю растекаться, я останавливаюсь и перезапускаюсь.
Сначала ясность.

## React и frontend-архитектура

1) **Как вы решаете между local и global state?**
Я начинаю с ownership и области совместного использования.
Если владеет один компонент — local state.
Если зависят несколько удалённых фич — global state.
Потом проверяю частоту обновлений и поведение при сбоях.

2) **Redux Toolkit или Zustand?**
Redux Toolkit — для явных transitions и team-scale audibility.
Zustand — для небольшого shared UI state с низкой «церемониальностью».
Выбираю по траектории сложности, а не по вкусу.

3) **Когда используете React Query?**
Для server state.
Cache, retries, окна устаревания и refetch-логика должны быть first-class.
Я не переизобретаю это через ad-hoc эффекты.

4) **Что вызывает hydration mismatches в Next.js?**
Недетерминированный render output, client-only API до mount и несогласованные fallback-деревья.
Сначала чиню детерминизм, потом улучшаю UX.

5) **SSR vs ISR vs CSR в одну строку?**
Критична свежесть: SSR.
Допустима устарелость и важен SEO: ISR.
Высокая персонализация и низкий приоритет SEO: CSR.

6) **Как снижаете render churn?**
Стабилизирую границы состояния, убираю лишние глобальные обновления, memoization — только после профилирования, уменьшаю шум идентичности props.
“No superstition. Measurement first.” [Без суеверий. Сначала измерения.]

7) **Ваш подход к frontend error handling?**
Граница за границей.
Безопасные fallback в UI.
Typed error contracts на API-границе.
Observability для triage.

8) **Как обрабатываете медленные списки?**
Keyset pagination, virtualization при необходимости, отмена запросов и стабильные item keys.
Потом измеряю p95 latency взаимодействия.

9) **Самый частый React anti-pattern, который всё ещё видите?**
Mirrored state между слоями.
Он создаёт расхождение и шум отладки.
Single source of truth не является опцией.

10) **Как объясняете state-стратегию интервьюерам не из frontend?**
Я объясняю это как ownership данных и containment отказов.
Где живёт истина.
Кто может её менять.
Какую устарелость мы готовы принять.

## Node.js, NestJS, API

11) **Как структурируете NestJS module?**
Controller для транспорта, service для use-case логики, repository для persistence.
Между ними явные контракты.
Никаких domain-правил в controllers.

12) **REST или GraphQL?**
REST — для понятных ресурсов и предсказуемого cache.
GraphQL — для гибкой композиции.
Если GraphQL, я настаиваю на resolver observability и query cost controls.

13) **Как предотвращаете дубли side effects?**
Idempotency keys на границах.
Единый слой retry-политики.
Consumer dedupe там, где есть события.

14) **Как проектируете систему под изменения?**
Сначала границы business capabilities.
Явные интерфейсы.
Backward-compatible эволюция.
Короткие feedback loops с продакшен-телеметрией.

15) **Как выстроена валидация?**
DTO validation на транспортной границе.
Domain invariants внутри бизнес-слоя.
Стабильное отображение известных отказов в error contracts.

16) **Ваш debugging workflow в production incidents?**
Воспроизвести сигнал.
Сопоставить logs, traces и metrics.
Сузить поверхность сбоя.
Выпустить минимально безопасный фикс.
Потом добавить regression-ограждения.

17) **Как избегаете over-engineering?**
Я привязываю решения к текущим ограничениям: размер команды, риск, горизонт.
Если сложность не даёт снижения риска в обозримом сроке, я откладываю её.

18) **Как думаете о retries?**
Неуправляемые retries — мультипликатор нагрузки.
Я централизую политику, добавляю jittered backoff и чёткие stop conditions.

19) **Что делает API «production-ready»?**
Понятные контракты, observability, rate limiting где нужно, idempotency для рискованных записей и rollback-safe путь деплоя.

20) **Как быстро ревьюите backend PR?**
Сначала correctness.
Потом failure modes, performance-эффекты и operability.
Стиль — в конце, если не мешает пониманию.

## System design и архитектура

21) **Monolith или microservices?**
Стартую с modular monolith, пока масштаб/ownership-ограничения не докажут обратное.
Делим, когда выгода выше налога распределённой системы.

22) **С чего начинаете ответ в system-design интервью?**
Уточняю метрику успеха и главное ограничение.
Потом даю baseline, риски и фазовую эволюцию.

23) **Как обсуждаете consistency trade-offs?**
Strict consistency для финансовой/юридической корректности.
Eventual consistency там, где это терпит пользователь и важнее throughput.
Дальше называю компенсирующие контроли.

24) **Как держите архитектуру практичной?**
Time.
Cost.
Risk.
Если trade-off нельзя квантифицировать, ответ не готов.

25) **Что оптимизируете первым: scalability или reliability?**
Reliability для ожидаемого масштаба.
Масштаб без надёжности — просто дорогой сбой.

26) **Как проектируете fallback paths?**
Явно определяю режимы деградации.
Частично работающий сервис должен оставаться полезным.
Fail closed или fail open выбираю по business risk, а не по инстинкту.

27) **Как ясно презентуете trade-offs?**
“We chose X because Y constraint mattered more than Z benefit, and accepted cost C.”
[Мы выбрали X, потому что ограничение Y было важнее выгоды Z, и приняли цену C.]
Коротко и с ответственностью.

28) **Как решаете архитектурные разногласия?**
Запрашиваю критерии решения, сравниваю опции по ограничениям, запускаю маленькие эксперименты там, где неопределённость велика.
Спор допустим.
Неопределённость — нет.

29) **Ваш архитектурный red flag?**
Скрытая связность без ownership.
Она замедляет каждое изменение и прячет пути отказа.

30) **Как понимаете, что дизайн готов?**
Критические пути тестируемы.
Failure modes известны.
Observability и rollback спланированы.
И команда уверенно это оперирует.

## AI-агенты, prompt engineering, AI-first delivery

31) **Как вы используете AI-агентов ежедневно?**
Упаковка контекста, role-split исполнение, маленькие diff, строгая верификация.
Ускорение черновой работы при сохранении человеческого суждения.

32) **Ваш главный AI anti-pattern?**
Просить большие переписывания без ограничений.
Это генерирует объём, а не уверенность.

33) **Как промпты уменьшают rework?**
Они превращают расплывчатые запросы в задачи с ограничениями и acceptance checks.
Это повышает точность первого прохода.

34) **Как контролируете hallucinations в инженерном потоке?**
Привязываю ответы к исходному контексту, требую явной неопределённости и проверяю через код/тесты до принятия.

35) **Как оцениваете качество agent output?**
Defect rate, rework rate, review cycle time и escaped issues после merge.
Если метрики не улучшаются, workflow меняю.

36) **Как держите контекст чистым в длинных сессиях?**
Периодические context reset.
Повторная фиксация инвариантов.
Новые промпты, привязанные к текущим code boundaries.

37) **Как объясняете AI-ценность руководству?**
Снижение cycle-time при стабильном качестве.
Не новизна модели.
Доказательство — delivery-метрики.

38) **Как безопасно используете RAG?**
Scoped retrieval, versioned content, grounded response contract и evaluation benchmarks до rollout.

39) **Как выбираете модели в продакшене?**
По ограничениям задачи: precision, latency, throughput и cost.
Универсального победителя нет.

40) **Какие решения вы никогда не делегируете агентам?**
Финальные architecture trade-offs, принятие риска и суждение в production incidents.
Это остаётся инженерной ответственностью.

## Delivery, leadership и поведенческие вопросы

41) **Как удерживаете стабильность релизов?**
Маленькие изменения, строгие gate, воспроизводимые артефакты, протестированные rollback-path и наблюдаемые rollout-метрики.

42) **Что делаете, если чего-то не знаете?**
Говорю это прямо.
Определяю короткий план закрытия.
Отчитываюсь по прогрессу с evidence.
Никакого defensive theatre.

43) **Опишите провал.**
Беру пример с ясным ownership.
Объясняю, что я упустил, что изменил и какой guardrail добавил, чтобы это не повторилось.

44) **Как приоритизируете под дедлайном?**
Сначала impact, потом reversibility, потом effort.
Если задачи равны, выбираю ту, что сильнее снижает будущий риск.

45) **Как менторите инженеров?**
Общие decision frameworks, дисциплина review и явное объяснение причин.
Я учу trade-off, а не только синтаксис.

46) **Как реагируете на жёсткую обратную связь?**
Воспринимаю как сигнал.
Извлекаю конкретные пункты.
Отвечаю конкретным изменением, а не спором.

47) **Как работаете с product managers?**
Выравниваю outcome-метрику, ограничения и допустимый риск.
Перевожу технические trade-offs в последствия для доставки.

48) **Как доказываете business impact?**
Связываю технические изменения с измеримыми результатами: latency, conversion, incident rate, cost, lead time.

49) **Что для вас означает seniority?**
Надёжное суждение под неопределённостью.
Стабильная доставка через других.
Ответственность за последствия, а не только за код.

50) **Почему мы должны нанять именно вас?**
Я доставляю full-stack системы с дисциплинированной архитектурой, практичным AI-agent workflow и спокойным исполнением под давлением.
Я ясно коммуницирую trade-offs.
Я снижаю риск, одновременно ускоряя доставку.

## Финальный drill-протокол, который я использую в последние 90 минут

Я провожу пять раундов.
По десять вопросов в раунде.
Таймер включён.

Поток раунда:

1. Ответ менее чем за 45 секунд.
2. Убрать filler-слова.
3. Добавить один конкретный production-пример.
4. Явно назвать один trade-off.
5. Завершить измеримым результатом.

Если ответ слаб, сразу переписываю и повторяю.
Без эго.
Только заточка.

К финалу я не звучу заученно.
Я звучу операционно.
Вот цель.


Для соседней глубины я фиксирую одно правило: качество интервью растёт через плотные evaluation loops — короткая попытка, чёткая оценка, немедленная коррекция и повтор, пока ответ не станет надёжным под давлением.
И долгосрочный карьерный вектор важен даже в интервью-подготовке, потому что сегодняшние ответы должны быть согласованы с траекторией навыков, которую вы планируете строить в ближайшие годы.
Если полезно, эта финальная глава естественно связана с [evaluation loops](../ai_about_ai/05_prompt_engineering_evaluation_loops.md) и [the next three years](../ai_about_ai/25_career_strategy_the_next_three_years.md): интервью улучшается, когда практика измерима, а стратегия — долгосрочная.
