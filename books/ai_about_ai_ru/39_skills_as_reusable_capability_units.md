# Skills как переиспользуемые capability-юниты

В агентных системах одни промпты не масштабируются.
Переиспользуемые skills — масштабируются.

Skill — это упакованный workflow:
контекст + инструкции + инструменты + контракт результата.

## Почему skills действительно важны

Без skills каждая задача изобретается заново.

Симптомы узнаваемые:

- плавающее качество,
- постоянная ручная сборка промптов,
- хрупкие handoff между ролями,
- медленный онбординг,
- скрытое «племенное» знание операторов.

Skills переводят лучший опыт в повторяемые операции.

## Анатомия сильного skill

Полезный skill задаёт:

- границы проблемы,
- условия запуска,
- обязательные входы,
- вызовы инструментов и их порядок,
- схему выхода,
- fallback-путь,
- границы разрешений.

Если один элемент отсутствует,
начинается дрейф исполнения.

## Жизненный цикл skill

Относитесь к skills как к продуктовому активу:

1. найдите повторяющуюся боль,
2. выделите рабочий процесс-победитель,
3. упакуйте его в формат skill,
4. проверьте на живых задачах,
5. версионируйте и улучшайте,
6. выводите слабые варианты из оборота.

Нет жизненного цикла — нет надёжности.

## Принцип проектирования: сначала узко

Широкий skill «делаю всё» выглядит мощно,
но под нагрузкой ломается первым.

Лучше так:

- одна работа,
- одна понятная точка входа,
- один надёжный формат выхода,
- один ответственный владелец.

Специализация повышает доверие.

## Governance и discoverability

Библиотеки skills разваливаются,
когда люди не могут найти нужный юнит.

Держите индекс, где есть:

- название skill,
- одно предложение use-case,
- требуемые разрешения,
- примеры запуска,
- известные ограничения,
- статус деприкации.

Обнаружимость — часть производительности.

## Метрики библиотеки skills

Считайте:

- частоту переиспользования,
- success rate,
- среднее время завершения,
- долю эскалаций,
- число инцидентов на версию skill,
- частоту rollback.

Если переиспользование низкое,
либо плохо с discoverability,
либо сам skill не попал в задачу.

## Анти-паттерны

- Skills, которым нужен скрытый контекст команды.
- Skills без контракта выхода.
- Skills, которые по умолчанию дёргают слишком много инструментов.
- Skills, которые не обновлялись после сбоев.
- Skills с правами шире, чем сама задача.

Это уже не библиотека.
Это музей.

## Практичный порядок сборки

Начните с трёх задач с самым высоким трением.
Сделайте по одному skill на задачу.
Мерьте результат две недели.
Потом решайте, что действительно расширять.

Вам не нужны пятьдесят skills.
Вам нужны десять, которым доверяют.

## Система возможностей, которая масштабируется

**Переиспользуемые AI skills** — это операционный слой между талантом модели и предсказуемым исполнением.

Проектируйте их узко.
Версионируйте осознанно.
Мерьте честно.
