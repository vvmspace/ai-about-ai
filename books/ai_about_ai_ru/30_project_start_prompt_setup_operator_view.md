# Стартовый промпт-сетап проекта (взгляд оператора)

Первый день проекта
редко ломается из-за нехватки идей.

Он ломается из-за неуправляемости.

Операторский взгляд на prompt setup
про одно:

предсказуемость,
контроль,
стоимость.

## Что оператор считает успехом

Не «красивый ответ модели».

А вот что:

- выход стабилен,
- риски видимы,
- стоимость сценария понятна,
- сбои обрабатываются штатно.

Если этого нет,
любой wow-результат
одноразовый.

## Старт-пакет: пять обязательных артефактов

Перед первым production-прогоном
должны существовать:

1. **Mission brief** — цель, пользователь, дедлайн.
2. **Constraints ledger** — правовые/технические/брендовые ограничения.
3. **Reference canon** — иерархия источников и уровень доверия.
4. **Output schema** — в каком формате «готово».
5. **Evaluation rubric** — как измеряем pass/fail.

Нет одного пункта —
получаете дрейф ещё до конца первой недели.

## Environment assumptions: блок, который часто забывают

Большая часть срывов
не в логике,
а в неоговоренной среде.

Явно фиксируйте:

- стек и версии,
- доступные инструменты,
- лимиты бюджета,
- запрещённые зависимости,
- целевую среду развертывания.

Один короткий блок
экономит много бессмысленных итераций.

## Трёхслойная архитектура промптов

### 1) System layer

Роль агента,
границы,
политики,
тон.

Меняется редко.

### 2) Task layer

Текущая цель,
критерии приёмки,
локальные ограничения.

Меняется по задаче.

### 3) State layer

Принятые решения,
открытые вопросы,
текущие риски.

Меняется постоянно.

Ключевая дисциплина:
фиксировать system,
версионировать task/state.

## Gate-логика: проверяем не только output, но и input

Типичная ошибка:
команда оценивает только ответы модели.

Сильная практика:
оценивать и качество самого запроса.

Минимум две оценки:

- **ambiguity score** — сколько трактовок допускает формулировка,
- **dependency score** — насколько результат хрупок к внешним фактам.

Если score высокий,
масштабировать исполнение рано.

## Дефолты на случай инцидента

Сразу заложите «спокойные» значения по умолчанию:

- фактологические утверждения с источниками,
- явная маркировка неопределённости,
- запрет категоричности в high-risk доменах,
- эскалация при конфликте ограничений.

Это снижает цену «уверенной чепухи».

## Операторские метрики первой недели

Отслеживайте:

- долю принятых результатов без ручной правки,
- среднее число итераций до приемки,
- стоимость на один принятый артефакт,
- частоту эскалаций,
- время до первого полезного результата.

Эти метрики показывают,
управляете ли вы системой,
или система управляет вами.

## Режим запуска: 15-минутный checklist

Перед стартом спринта:

- mission и constraints подтверждены,
- schema и rubric доступны исполнителю,
- source hierarchy задокументирована,
- escalation path назначен,
- dry-run прогнан и оценён по рубрике.

Пять пунктов.
Они окупаются очень быстро.

## Типовые анти-паттерны старта

1. «Давайте сначала быстро попробуем без формальностей».
2. Смешивание system и task инструкций в один слой.
3. Неявная стоимость ошибок.
4. Отсутствие fallback-плана при отказе инструмента.
5. Нулевое логирование причин отклонения результата.

Каждый пункт
выглядит невинно,
пока не накопится технический долг процессов.

## Как выглядит зрелый старт

- Правила короткие,
  но исполнимые.
- Рубрика проста,
  но жёсткая.
- Риски не замалчиваются,
  а маршрутизируются.
- Любая новая задача
  встраивается в существующую операционную рамку.

Это и есть масштабируемость.

## Главная мысль

Стартовый промпт-сетап оператора —
это не «написать хороший промпт».

Это построить контрольную поверхность,
через которую проект проходит
без героизма и паники.
